# RFC — 0.3.0: Crash-safe reconciliation & recovery

> **Status: IMPLEMENTED (Phases 0–2).** Phase 0 (schema + enums), Phase 1
> (write-ahead marker), and Phase 2 (marker-aware recovery, the `reconcile` /
> `recover` / `unlock` commands, deploy gating, and the `doctor` `next_action`
> additions) have landed. See [cli.md](../cli.md) for the shipped command
> surface. This document remains the design of record.
>
> **Scope: Phases 0–2** (§9); design decisions locked 2026-07-20 (§10). Phase 3
> items — the `refs/mergetrain/deploys/*` rewrite-guard ref and `locks.host_id`
> liveness — are **deferred** out of 0.3.0.

**Overriding invariant:** after *any* crash, mergetrain must reconcile local DB
state with the real remote git state and **never lie** — never mark a job
`deployed` unless a configured push ref actually carries the sha, and never
requeue (re-push) a job whose push already landed.

## 1. The problem

The one irreversible step in a deploy is `git push --atomic`. Between that push
and the final `mark_job(deployed)` commit there is a window where a crash leaves
the remote **advanced** but the DB still saying `in_progress`. Today's recovery
(`_requeue_orphans`) blindly resets orphaned `in_progress` jobs to `queued`,
which would **re-push an already-landed deploy** — or, symmetrically, a lost
pre-push marker under `synchronous=NORMAL` could let a power loss drop the
record while the remote moved. The push outcome is currently only ever held in
memory (`push_status='succeeded'` at git_runner.py:1520) and on the singleton
`locks.head_sha`, which DEAD-owner recovery **deletes before reconcile could
read it** (store.py:414).

## 2. Design basis

Three approaches were designed and adversarially critiqued (kill-at-phase, "does
it stay truthful?"):

| Approach | Untruthful kill-points | Verdict |
|---|---|---|
| **Idempotent replay + durable per-job marker** (chosen backbone) | **0 / 10** | fixable |
| Write-ahead deploy-intent table (WADI) | 0 / 10 | fixable |
| Remote-as-truth, zero new schema | **3 / 12** | fixable |

**Chosen: a hybrid, backbone = durable per-job marker.** The marker lives on the
`deploy_queue` *row*, so it survives the `locks` row deletion that DEAD-owner
recovery performs — the structural flaw that sinks the zero-schema approach
(its only evidence, `locks.head_sha`, is deleted before reconcile runs). From
the remote-as-truth approach we adopt **remote refs as the sole authority** for
"did the push land" and **`PRAGMA synchronous=FULL` before the push** so the
marker is fsync-durable before the network side effect. From WADI we adopt
**write-ahead the marker in one `IMMEDIATE` txn strictly before the remote is
touched** and an **immutable audit ref** to survive a later remote
history-rewrite. WADI's separate `deploy_intents` table was rejected as
over-schema (everything fits on `deploy_queue`).

## 3. Schema & durability changes (migration 6, `SCHEMA_VERSION` 5→6)

All additive; pure `ALTER ADD COLUMN`, no backfill.

| Change | Purpose | Truthfulness-critical |
|---|---|---|
| `deploy_queue.pending_deploy_sha TEXT NOT NULL DEFAULT ''` | The exact commit the runner is about to atomically push. Written with `push_status='pending'` in a `synchronous=FULL IMMEDIATE` txn immediately before push at **both** push sites (git_runner.py:986 and :1506). A non-empty value on an `in_progress`/`needs_reconcile` row is the *sole durable proof* a push was attempted; reconcile tests it against the remote. Cleared on `deployed`/`canceled`/`queued`; **preserved on `blocked`** for forensics. | **Yes** |
| `status` enum: add **`needs_reconcile`** to `ACTIVE_STATUSES` | Non-terminal parking state. `_requeue_orphans` sends marker-bearing orphans here instead of `queued`, so a possibly-pushed train is never silently re-deployed. Membership in `ACTIVE_STATUSES` keeps `_active_branch_count` blocking duplicate enqueue while reconcile is pending. | **Yes** |
| `push_status` enum: add **`pending`** | Durable "push attempted, outcome unknown", distinct from `not_run`/`succeeded`/`failed`. | **Yes** |
| `verify_status` enum: add **`unknown`** | reconcile can prove the deploy landed but **cannot** prove post-push verify hooks ran. `unknown` is the only non-lying value; reconcile never re-runs verify against a now-stale sha. `mark_job` (store.py:917) rejects values outside the set, so this is required or every path crashes. | **Yes** |
| git ref `refs/mergetrain/pending/<job_id>` = pending sha | Pins the pending commit in the object store so `git merge-base --is-ancestor` resolves even after an external `git gc --prune=now` removes the crashed worktree's dangling commit. | Yes (durability) |
| git ref `refs/mergetrain/deploys/<deploy_sha>` (pushed inside the same atomic refspec list) | Rewrite guard: if a human later force-pushes a payload ref backward past a landed sha, reconcile still finds the sha under the audit ref and routes to `blocked` instead of re-deploying over the rollback. | **Deferred** (Phase 3) |
| `locks.host_id TEXT NOT NULL DEFAULT ''` (`hostname:boot_id`) | Availability only: lets `owner_liveness` treat a matching pid on a different boot as DEAD, cutting PID-reuse false-ALIVE wedges. Truthfulness holds without it. macOS: `sysctl kern.boottime`; degrades to pid-only. | **Deferred** (Phase 3) |
| `PRAGMA synchronous=FULL` on the runner connection (verified missing today — connect() sets only WAL) | Under default `NORMAL`, a power loss can lose the pre-push marker commit while the push already advanced the remote → re-deploy. FULL makes "no marker" truly prove "never pushed". | **Yes** |

## 4. Crash state machine

| Phase | On-disk marker after crash | On-restart resolution |
|---|---|---|
| **P0** claim → assembly → gates, before marker txn | `pending_deploy_sha=''`, remote unchanged | Empty marker ⇒ **CLEAN** ⇒ reset to `queued` offline, no git call (today's fast path — still correct, nothing was pushed). |
| **P1** marker committed, `git push` in flight / died mid-invocation | `pending_deploy_sha=P`, `push_status='pending'`; remote **either** unchanged **or** fully advanced (atomic push is all-or-nothing per invocation) | Orphan → `needs_reconcile`. reconcile: all push_refs contain P ⇒ `deployed`; none ⇒ `queued`; mixed ⇒ `blocked`. |
| **P2** push returned 0, before final mark_job | `pending_deploy_sha=P`, `push_status` still `pending`; remote at P; `pushing/success` event committed | reconcile: every push_ref contains P ⇒ finalize `deployed`, `deploy_sha=P`, `push_status='succeeded'`, `verify_status='unknown'`. Never requeued. **(principal gap today.)** |
| **P3** post-push verify running/failed, before mark_job | `pending_deploy_sha=P`, remote at P, verify not persisted | reconcile: remote has P ⇒ `deployed` + `verify_status='unknown'`, `next_action=verify_reconciled_deploy`. Verify **not** re-run. |
| **P4** final mark_job committed, before cleanup/release | `status='deployed'`, marker cleared, pending ref deleted; leftover worktree + stale lock | DEAD-owner acquire deletes lock, reconcile no-op. Worktree is a gc candidate. Only cruft. |
| **P5** wedge: expired lock, owner ALIVE/UNKNOWN, `in_progress` jobs | expired lock, `owner_liveness` ALIVE/UNKNOWN | `_acquire_runner_lock` still raises `LockHeld` (store.py:418-421) — never auto-steals from a possibly-live push. `next_action=unlock_wedged_runner` → operator runs `unlock --force`. |
| **P6** cancel raced the push window (`cancel_requested_at` **and** marker set) | `pending_deploy_sha=P`, `cancel_requested_at!=''` | → `needs_reconcile` (not straight `canceled`): the remote decides. P landed ⇒ `deployed` (note records the ignored late cancel); not landed ⇒ `canceled`. Cancel-pending **without** a marker is still safe to cancel offline. |

## 5. Reconcile decision table

For each `needs_reconcile` job, recover P from `pending_deploy_sha`, then
`git fetch` + `git ls-remote <remote> <push_refs>` + `refs/mergetrain/deploys/*`
and test with `git merge-base --is-ancestor P <ref>`:

| DB | Remote vs P | Resolution |
|---|---|---|
| `needs_reconcile`, marker=P | **every** push_ref contains P | `deployed` (`deploy_sha=P`, `push_status=succeeded`, `verify_status=unknown`) |
| `needs_reconcile`, marker=P | **no** push_ref contains P, P still resolvable | `queued` (requeue; clear marker, delete pending ref) |
| `needs_reconcile`, marker=P | **mixed** across multiple push_refs | `blocked` (human decision) |
| `needs_reconcile`, marker=P | P unresolvable (pin ref gone, pruned) | `blocked` (refuse to guess) |
| any | remote unreachable | **no-op**, exit 7, jobs stay `needs_reconcile` |

> **Deferred (Phase 3):** *landed-then-rewound* detection — P landed, then a push
> ref was force-pushed backward before reconcile ran — requires the
> `refs/mergetrain/deploys/*` audit ref. Until it ships, that (rare, 3-way)
> case would requeue rather than block, so the operational rule is: **run
> `reconcile` before any manual force-push after a crash.**

## 6. Command surface

All three read state first, acquire the runner lock (serialized against
`run-batch`), and **never push** — the only writes are the truthful
finalize/requeue/block a crash-free run would also have produced.

### `mergetrain reconcile [--apply] [--json]`
Resolve every `needs_reconcile` job against the remote. Default dry-run.
- **JSON**: per-train `{train_id, job_ids, pending_deploy_sha, push_refs:[{ref, remote_sha, verdict}], audit_ref_present, decision, applied, deploy_sha, push_status, verify_status}` + `summary{reconciled_deployed, requeued, conflicts, skipped_live}` + `next_action`.
- **Exit**: `0` resolved/nothing to do · `2` usage/config · `3` lock held by live owner (retryable) · `7` remote unreachable (nothing changed) · `10` ≥1 job `blocked` (needs human).

### `mergetrain recover [--gc] [--json]`
One-button restart heal for phone/agent use: split orphans (empty marker→`queued`, marker→`needs_reconcile`, cancel-without-marker→`canceled`), optionally gc crashed worktrees, then `reconcile --apply`. **Never** ships queued/validated work (no deploy as a side effect).
- **Exit**: `0` recovered clean · `2` usage · `3` live owner holds lock · `7` remote unreachable · `10` conflicts remain.

### `mergetrain unlock [--force] [--json]`
Clear a wedged lock (the P5 expired-but-ALIVE/UNKNOWN + `in_progress` branch).
Without `--force`: only deletes a DEAD/absent owner's lock. With `--force`,
**ordering is load-bearing**: (1) classify against the remote **first** — if
unreachable, abort exit 7, change nothing; (2) only then delete the lock and run
the marker-aware split. It **never** itself writes `deployed`/`failed` — that
verdict comes only from the subsequent remote-verified reconcile. Appends an
append-only `run_events` audit row (`phase='unlock'`, `state='forced'`) capturing
prior owner, liveness verdict, host_id, token, lease times, operator identity,
the wedge reason, and the pre-release reconcile summary; echoes `audit_event_id`.
- **Exit**: `0` cleared · `2` usage · `4` refused (owner ALIVE, no `--force`) · `5` no lock · `7` remote unreachable during classification.

### Deploy gating (decisions Q4 + Q7)

While any job is `needs_reconcile`, `run-batch --deploy` **refuses** (exit +
`next_action=reconcile_pending_deploy`): deploys target the same push_refs, so a
new push must not run before reconcile resolves the pending state. `run-batch`
also **auto-classifies** orphans at startup (marker-aware split, detect + report
only) — it never auto-mutates deployed/failed state, so recovery stays an
explicit `reconcile`/`recover` action.

## 7. `doctor` `next_action` additions (DB-only, no remote call)

| Value | When emitted |
|---|---|
| `reconcile_pending_deploy` | ≥1 job in `needs_reconcile` (or a DEAD-owner orphan with non-empty marker). |
| `unlock_wedged_runner` | Lock expired, owner ALIVE/UNKNOWN, `in_progress` jobs exist (the store.py:418-421 wedge). |
| `verify_reconciled_deploy` | A job was finalized `deployed` with `verify_status='unknown'` — re-run verification manually; deploy is not re-pushed. |
| `reconcile_conflict_manual` | reconcile left ≥1 job `blocked`; human git inspection required (falls into existing `fix_blocked_job`). |

## 8. Fault-injection test matrix (kill-at-phase → expected post-recovery state)

- **kill_pre_marker_requeues_offline** — SIGKILL after merge, before marker txn ⇒ empty marker, requeue to `queued`, no network call, remote unchanged.
- **kill_after_marker_before_push_no_remote_move** — ⇒ `needs_reconcile`, pending ref present; reconcile finds no ref contains P ⇒ requeue; no re-push.
- **kill_mid_atomic_push** (applied vs rejected variants) — ⇒ same on-disk marker; the remote alone disambiguates `deployed` vs requeue.
- **kill_after_push_before_final_mark** — ⇒ `needs_reconcile`, `push_status='pending'`; reconcile ⇒ `deployed`, upgraded to `succeeded`, no duplicate push. *(principal-gap regression.)*
- **powerloss_after_push_marker_durable** — under FULL the marker survives ⇒ `deployed`; companion assertion under NORMAL asserts it *misclassifies* (guards the pragma).
- **kill_in_verify_window** — ⇒ `deployed`, `verify_status='unknown'`, `next_action=verify_reconciled_deploy`; verify not re-run.
- **kill_after_final_mark_leaves_only_cruft** — ⇒ `deployed`, marker cleared; only a stale lock + worktree cruft.
- **process_one_isolation_push_marked** — force the gate-fail one-by-one isolation path and kill at git_runner.py:986 ⇒ marker set (proves **both** push sites are instrumented). *(Design 3's central gap.)*
- **cancel_plus_marker_defers_to_remote** — cancel during gates, kill after marker + after push lands ⇒ `needs_reconcile` → `deployed` with note; twin (not landed) ⇒ `canceled`.
- **pending_sha_pruned_then_reconcile** — `git gc --prune=now` then reconcile ⇒ pin ref keeps P resolvable; variant deleting the pin ⇒ `blocked` (no guess).
- **landed_then_rewound_blocks** — land, crash, force-push payload backward, reconcile ⇒ audit ref carries P ⇒ `blocked`, no re-deploy over rollback.
- **remote_unreachable_is_noop** — reconcile with unreachable remote ⇒ exit 7, zero DB mutations, jobs intact.

## 9. Implementation phases

- **Phase 0 — durability + enums (no behavior change).** `synchronous=FULL`; add the three enum values; audit every reader of the closed sets (`snapshot.next_action`, status renderers, `_active_branch_count`); migration 6. *Risk: low; main risk is a missed enumerator — grep every use of the three tuples.*
- **Phase 1 — write the marker at BOTH push sites.** One shared pre-push helper writes `pending_deploy_sha`+`push_status='pending'` for exactly `merged_jobs` and creates the pin ref, then pushes; wire into process_batch (:1506) **and** process_one (:986). Extend `mark_job` to clear the marker + delete the pin on `deployed`/`canceled`/`queued`, **preserve on `blocked`**. *Risk: medium; the classic gap is forgetting the isolation push site — covered by a test.*
- **Phase 2 — marker-aware recovery + reconcile engine + doctor.** Split `_requeue_orphans`; build the reconcile classifier (`ls-remote` + `merge-base --is-ancestor`); ship `reconcile`/`recover`/`unlock`; extend `doctor --json`. *Risk: medium-high; `needs_reconcile` is a new refusal state callers must handle, and a restart with an unreachable remote now parks jobs instead of auto-requeuing (deliberate). Fault matrix is the gate.*
- **Phase 3 — rewrite guard + liveness hardening — DEFERRED out of 0.3.0 (independently shippable later).** Push `refs/mergetrain/deploys/<sha>` in the atomic push and consult it in the not-landed branch; populate `locks.host_id` and rework `owner_liveness`. *Risk: low-medium; keep the pid-only fallback (boot_id was a prior source of flaky macOS tests).*

## 10. Resolved decisions (locked 2026-07-20)

1. **No-op train** (assembled HEAD already == remote tip): **accept as benign** — if a push ref carries the sha, `deployed` is truthful regardless of which runner moved it. A `note` records "sha already live (no-op)". No `pushing/success`-event gate.
2. **Rewrite guard** (`refs/mergetrain/deploys/*`): **deferred out of 0.3.0** (Phase 3). Without it, a landed-then-rewound-during-recovery would requeue rather than block; mitigated operationally — **run `reconcile` before any manual force-push after a crash** (documented as a known limitation, §5).
3. **`synchronous=FULL`**: **global**, on the runner connection. Deploys are infrequent; one fsync/commit is negligible and removes the "forgot the pre-push checkpoint" failure mode.
4. **Deploy while `needs_reconcile` pending**: **hard block**. `run-batch --deploy` refuses while any `needs_reconcile` job exists (exit + `next_action=reconcile_pending_deploy`) — deploys target the same push_refs and must not muddy a pending reconcile (§6 *Deploy gating*).
5. **`locks.host_id` / boot-id liveness**: **deferred out of 0.3.0** (Phase 3) — availability-only (truthfulness holds without it), and boot_id was a prior flaky-macOS source. Ship pid-only liveness; the wedge case is handled by `unlock --force`.
6. **gc of mergetrain refs**: **`gc --apply` sweeps stale `refs/mergetrain/pending/*`** (delete once the owning job is terminal). `deploys/*` retention is moot until the Phase-3 audit ref ships.
7. **Startup reconcile**: **`run-batch` auto-classifies orphans at startup (detect + report only); DB mutation requires an explicit `reconcile`/`recover`.** Consistent with (4) and the "explicit intent" ethos.

**Net 0.3.0 scope: Phases 0–2.** Phase 3 (rewrite-guard audit ref + `host_id` liveness) is deferred to a later release.
