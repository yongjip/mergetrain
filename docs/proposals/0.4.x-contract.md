# RFC 0.4.x — Versioned JSON/config contract + migration preflight (issue #44)

Status: proposed. Supersedes ad-hoc, versionless JSON. Last 1.0-gate item; directly precedes the 0.9.0 API-FREEZE RC.

## 0. Problem & thesis

Today no machine-readable surface carries a contract number. The only `"version"` fields — `doctor` (cli.py:545) and `version` (cli.py:218) — are the PRODUCT `__version__`, which bumps for unrelated reasons. 27 distinct JSON/JSONL payload shapes are versionless, and the product ethos is "agents read JSON and act instead of guessing" (CLAUDE.md), so a silently-changed shape is a real hazard.

Two precedents already exist to mirror: `store.SCHEMA_VERSION=7` with `PRAGMA user_version` + an `ensure_schema` gate (store.py:28,129-135), and `registry.REGISTRY_VERSION=1` embedded in `repos.json` (registry.py:27,118). The registry is the anti-pattern to avoid: it *writes* a version but `load_registry` never *reads* it (registry.py:85-112). Every version we emit must be paired with a load-time check.

Thesis: freeze a frame worth freezing. Fix the known-incoherent envelope in the same PR that establishes contract 1, stamp it in exactly one place per serializer, and back the freeze with a CI forcing function — not a doc.

## 1. Two version numbers, both distinct from the product version

- `CONTRACT_VERSION = 1` — new module `src/mergetrain/contract.py`. Governs the shape of every machine-readable emission (one-shot JSON, HTTP snapshot, JSONL stream). Importable by cli.py / snapshot.py / hub.py / observability.py with no circular import.
- `CONFIG_VERSION = 1` — new constant near the top of `src/mergetrain/config.py`. The schema version of `.mergetrain.yaml`.

Both are integers (matching `SCHEMA_VERSION` / `REGISTRY_VERSION` / `user_version`, each one-number-per-artifact). `__version__` stays untouched; `doctor`/`version` keep `"version": __version__` and gain a SEPARATE `contract_version`. A patch release never reads as a contract change.

**Why one global int, not per-surface (rejecting Philosophy B):** the surfaces are not independent — `hub status` embeds a full `build_dashboard_snapshot` per repo (hub.py:198); `status`/`inspect`/`run-batch --preview` embed `job.to_dict()`/`_results_payload` (cli.py:629). Per-surface numbers nest and contradict inside one payload, and force a consumer to track an N-way matrix for one product. Philosophy B could not even hold its own line — it collapsed ~13 payloads (including the freshly-shipped, still-evolving recover/reconcile/unlock recovery surface) into one `RESULT_CONTRACT`, reproducing the global coupling it condemned. A single top-level integer is the honest granularity.

## 2. Fix the frame BEFORE stamping it (contract-1 baseline)

`#44` is the last moment before the freeze where breaking today's shape is free — there is no prior contract number to break. We normalize the three defects the inventory proves make the envelope untrustworthy, then stamp that fixed shape as contract 1.

**2a. `ok` becomes uniform = "the command executed and produced its result without raising an error envelope."** Today `ok` is overloaded four ways: constant-true in most; a health verdict in `doctor` (cli.py:569); `result=="success"` in `_results_payload` so a completed run with post-push verify warnings returns `ok:false` (cli.py:640-649); found-or-not in `hub remove` (cli.py:1183); and absent entirely from `agent-contract` (cli.py:110). Fix:
- `doctor`: keep computing the config_exists-AND-repo_root verdict but move it to a new `health` field; set `ok:true` (the command ran).
- `_results_payload`: set `ok:true` whenever the run completed and emitted a result; the graded outcome stays in the existing `result` field (`success`/`warning`/`partial`/`failed`). A completed-with-warnings run is `ok:true, result:"warning"`. Consumers branch on `result`, never on `ok`, for outcome.
- `hub remove`: `ok:true` for a completed removal attempt; the existing `removed` bool carries found-or-not.
- `agent-contract`: add `ok:true`.

After 2a, `ok` means exactly one thing everywhere: no error envelope present.

**2b. Add `next_action` to `status`** (cli.py:286-296). CLAUDE.md tells agents to read `status --json` OR `doctor --json` before acting, yet `next_action` is present in doctor/reconcile/recover/unlock/deploy-block/dashboard-snapshot but ABSENT from status. Make the two mandated reads symmetric.

**2c. Unify the three failure shapes into one envelope.** Today: (a) `main()` `{ok:false, error:{code,message,retryable}}` with `retryable` from exception TYPE (cli.py:1468-1485); (b) `_emit_recovery_error` same keys, `retryable` from EXIT_CODE in (3,7) (cli.py:908-923); (c) `_emit_deploy_reconcile_block` `{ok:false, result:"blocked", blocked_reason, next_action, note}` with NO `error{}` (cli.py:597-615). Collapse all failures to `{ok:false, error:{code,message,retryable}, next_action?}`. The deploy-block keeps its `next_action` and `blocked_reason` (moved under `error.code:"reconcile_pending_deploy"` + top-level `next_action`); recovery errors keep named codes. One failure shape, one place to parse.

These are the ONLY behavioral breaks, and they are deliberate: they are what a real coding-agent consumer needs before a freeze. Everything after this is additive.

## 3. Where the contract number lives

**One-shot JSON — stamp at the single serializer.** `dump_json` (cli.py:102) is the choke point for all ~21 one-shot sites, verified to include all three (now unified) failure paths. Change it to inject the field top-level:

```python
def dump_json(payload):
    if isinstance(payload, dict) and "contract_version" not in payload:
        payload = {"contract_version": CONTRACT_VERSION, **payload}
    print(json.dumps(payload, ensure_ascii=False, indent=2, sort_keys=True))
```

`sort_keys=True` places it deterministically. This covers `init` (cli.py:202, which emits unconditionally with no `--json` flag) and `agent-contract` (the payload with no `ok`) for free — the envelope field is orthogonal to `ok`. Embedded sub-objects (`job.to_dict()`, the dashboard snapshot nested in `hub status`) are deliberately NOT stamped; the outer frame owns the number.

Key name is `contract_version` (int), NOT bare `contract` — `contract` collides head-on with the `agent-contract` command whose payload literally *is* the operating contract; a numeric `contract:1` beside its rules-contract semantics would confuse agents on the one surface where the word already means something else.

**HTTP snapshot — stamp at its own boundary, and resolve the hub contradiction cleanly.** `build_dashboard_snapshot` (snapshot.py:249) is served raw by `dashboard.py /api/snapshot`, not via `dump_json`. Rule: `build_dashboard_snapshot` returns WITHOUT `contract_version`; the `/api/snapshot` HTTP handler adds it at emit time. Therefore the per-repo snapshots embedded inside `hub status` (hub.py:198) are the bare, unstamped dicts, and the hub's own top-level `contract_version` (added by `dump_json`) is the single authority. No hand-strip of an inner key (Philosophy A's self-contradiction and per-site surgery), and no fictional hub cross-version detection: the hub builds every embedded snapshot with its OWN binary (hub.py:198), so there is never a cross-version JSON mismatch to detect. The only genuine cross-version surface the hub touches is the DB schema, already enforced strictly by `store.connect(read_only=True)` (store.py:90-95), and a newer-config repo already degrades to an `ok:false` card via `_repo_entry`'s broad `except` (hub.py:153). We add nothing there.

**Streaming JSONL — a re-emitted stream header, records byte-identical.** Add one header line before the first event via `_dump_jsonl` (cli.py:306), a new discriminated type:

```
{"type":"stream_start","contract_version":1,"after_event_id":<n>}
```

`event`/`heartbeat`/`stream_end` records (observability.py:250-323) stay byte-for-byte unchanged, so `--after` integer-id resume is untouched. Critically, the header is re-emitted on EVERY (re)connect including `--after` resume — because a resume is a fresh process (cli.py re-enters `cmd_events`) that may be a DIFFERENT binary version, which is exactly the moment a long-lived consumer must re-confirm the contract. The header carries no event `id`, so it never collides with id-based dedupe. `stream_start` is a new record type; the compatibility rule (Section 5) requires consumers to dispatch on the existing `type` discriminator, which they already do for three types.

## 4. Config versioning + preflight

**File.** Optional top-level `version:` int in `.mergetrain.yaml` (bare scalar, inside the zero-dep `_parse_simple_yaml` subset). `default_config_dict` (config.py:265) and `render_default_config` (config.py:291) emit `version: 1`, so `mergetrain init` writes it going forward.

**Read/validate site.** Top of `load_config`, immediately after `data` is obtained and BEFORE section parsing (between config.py:443 and 445):

```python
raw = data.get("version")
config_version = 1 if raw is None else _positive_int(data, "version")  # reuse config.py:401
if config_version < CONFIG_VERSION:
    data = _migrate_config(data, config_version)   # forward-only, in-memory; empty at v1
    config_version = CONFIG_VERSION
# NOTE: too-new is NOT raised here — see enforcement scope below
```

- Absent ⇒ v1, never raises. Every `.mergetrain.yaml` in the wild (including this repo's own) has no `version` key; unknown keys are silently ignored today (config.py has no allowlist), so absent MUST map to current and fail-closed-on-absent would brick every install. This mirrors `config_exists=False` already falling back to `default_config_dict` without error.
- `_migrate_config(data, from)` is a dict-in/dict-out registry keyed by target version, exact analog of `store.py:212-255`'s `migrations` dict but over the parsed mapping. Empty (identity) at v1; the structure exists so v2 has a home.
- Thread `config_version:int` into `MergetrainConfig` (config.py:122-143) and emit it in `to_dict()` (config.py:136-143) — the ONLY serialization boundary to config JSON consumers (`inspect --json`); if it isn't wired there it never reaches agents.
- config.py raises `ConfigError` (not QueueError — the task framing was wrong); it flows through `main()`'s unified envelope (cli.py:1468) → `{"contract_version":1,"ok":false,"error":{"code":"config_error",...}}` for free.

**Enforcement scope — the recovery-permissive fix.** A too-new config (an older binary reading a file a newer binary wrote) must NOT hard-fail the crash-recovery commands — that is 0.3.0's flagship feature, and a version mismatch after a rollback must not lock the operator out of `reconcile`/`recover`/`unlock`. Therefore refuse-newer is enforced by a command-scoped guard, NOT inside `load_config`:
- `enqueue`, `run-batch`/`run-next` (the deploy/state-shipping path): `_preflight_guard` raises `ConfigError` when `config_version > CONFIG_VERSION`. Fail-closed — never ship code against a config an old binary may misread.
- `reconcile`, `recover`, `unlock`, and all read-only commands (`status`, `doctor`, `inspect`, `version`, `gc` dry-run): proceed; the mismatch is surfaced as a warning in the payload and as `doctor.next_action = upgrade_mergetrain`, but the command runs. Recovery stays the most permissive surface.

**No persist path in #44.** We do NOT ship `preflight --migrate`. `_parse_simple_yaml` cannot round-trip comments, and re-serializing hand-edited YAML from the frozen dataclass would destroy comments and silently-dropped unknown keys (a data-loss trap). At v1 there is nothing to migrate; `init` writes `version:1` for new files, existing files ride absent⇒1. If a future v2 needs persistence, it must do a targeted line-insert preserving the raw file, never a dataclass re-serialize.

## 5. Compatibility policy

- **Additive-only within contract N.** Adding a key to any payload, a new optional config key, a new event field, or a new command is NOT a bump. Consumers MUST ignore unknown keys and MUST dispatch JSONL on `type`.
- **Breaking = bump `CONTRACT_VERSION`.** Removing/renaming a key, changing a value's type or meaning, changing `ok`/`result` semantics, changing exit codes, or changing the JSONL frame/resume model. After 2a-2c, contract 1 has NO known-hostile field to defer — the `ok` cleanup rides the baseline, so we are not freezing a defect.
- **Config:** additive optional keys don't bump `CONFIG_VERSION` (unknown keys already tolerated). Bump only when an existing key's meaning changes or a key becomes required — i.e. when an old binary would misread a new file. Forward-only, no down-migration (inherited from store.py's one-way property).
- **Negotiation — one source of truth.** Agents read `contract_version` from `doctor` (the mandated first read) and `version`. No duplicate surface map, no `contract` subcommand, no `--require`, no new exit code — there is no second consumer that pins a major yet (YAGNI; defer until one exists). "I understand contract N" is a single integer comparison.
- **0.9.0 freeze linkage:** "`CONTRACT_VERSION` frozen at 1; only additive changes until a deliberate 2, enforced by the fingerprint gate."

## 6. The forcing function — golden key-set fingerprint CI gate

This is what converts "interfaces only change with a version bump" from a doc into a guarantee. JSON has no physical enforcement like SQLite refusing a mismatched `user_version`; the registry.py write-without-check gap is exactly what happens without one.

`tests/test_contract_fingerprints.py` + checked-in `tests/contract_fingerprints.json`:
- For every `--json` command and each JSONL frame type, capture a recursive KEY-SET fingerprint (keys only, values ignored, sorted, nested objects included) of a representative payload.
- Store the expected fingerprints keyed by `CONTRACT_VERSION` next to the constant's intent.
- The test FAILS whenever a live fingerprint diverges from the checked-in one. The failure message classifies mechanically: keys only ADDED ⇒ additive, permitted within contract N — regenerate the golden file and note it in the changelog; keys REMOVED/RENAMED ⇒ breaking — requires a `CONTRACT_VERSION` bump. Type/semantic changes it cannot auto-detect, but every shape change forces a conscious human decision and a golden-file regen.
- The test also asserts: every one-shot payload carries top-level `contract_version == CONTRACT_VERSION`; nested snapshots/job dicts do NOT; `stream_start` carries it and `event`/`heartbeat`/`stream_end` do not; an absent-version config loads as current; a `version: 999` config raises `config_error` on the deploy path but lets `recover` run.

This is the mechanism that makes the freeze real: any future PR that changes a payload shape without bumping the number fails CI. Adopt it regardless of granularity.

## 8. Implementation phases
- Phase 0 — foundation, zero behavior change. Add src/mergetrain/contract.py (CONTRACT_VERSION=1) and CONFIG_VERSION=1 in config.py. Add config_version:int to MergetrainConfig (config.py:122-143) + emit in to_dict(); add the load_config read/migrate-older gate (config.py:~444) with an empty _migrate_config; write version:1 in default_config_dict/render_default_config. NO enforcement yet. RISK: low — absent⇒1 makes it a no-op for every existing file including this repo's; the only trap is forgetting to wire config_version through to_dict() (then it never reaches inspect --json). Tests: existing configs load unchanged; version:1 == absent; version:999 is recorded but not yet enforced.
- Phase 1 — fix the frame (the only breaking phase; lands deliberately). Normalize ok to uniform command-executed semantics: move doctor's health verdict to a `health` field (cli.py:569), set _results_payload ok=true with outcome in `result` (cli.py:640-649), set hub-remove ok=true keeping `removed` (cli.py:1183), add ok:true to agent-contract. Add next_action to status (cli.py:286-296). Unify the three failure shapes (cli.py:597-615 / 908-923 / 1468-1485) into one {ok:false,error{code,message,retryable},next_action?} envelope. RISK: HIGH — this changes fields real consumers read; the hub dashboard and any agent branching on ok must be updated in lockstep. Mitigation: land it as one reviewed PR with all golden tests regenerated and a migration note; do it before any contract_version is visible so no consumer has yet pinned the old shape.
- Phase 2 — stamp the number (small, mechanical). Inject contract_version at dump_json (cli.py:102) — covers all ~21 one-shot sites including init and agent-contract. Add it at the /api/snapshot HTTP boundary (dashboard.py) while build_dashboard_snapshot (snapshot.py:249) stays bare, so hub-embedded snapshots carry no inner number. Add the re-emitted stream_start header via _dump_jsonl (cli.py:306); leave event/heartbeat/stream_end byte-identical. Add contract_version to doctor and version payloads. RISK: low-medium — the one subtlety is the snapshot boundary rule (builder bare, handler stamps); a fingerprint test (Phase 3) catches a leaked inner number. stream_start is a new JSONL type — document that consumers dispatch on `type`.
- Phase 3 — the forcing function + config enforcement. Ship tests/test_contract_fingerprints.py + tests/contract_fingerprints.json asserting every --json surface and each JSONL frame carries the right (or intentionally absent) contract_version and matches a checked-in key-set fingerprint, failing CI on any un-bumped shape change. Wire _preflight_guard on enqueue/run-batch/run-next only (fail-closed on config_version>CONFIG_VERSION); leave recover/reconcile/unlock and read-only commands permissive with a warning + doctor.next_action=upgrade_mergetrain. RISK: medium — the fingerprint gate must capture nested key-sets correctly or it under-enforces; the guard must be scoped to exactly the ship-code commands or it regresses the recovery-permissiveness decision. Tests: version:999 config fails enqueue with config_error but lets recover run.
- Phase 4 — docs + freeze note. Write docs/contract.md enumerating every surface, its contract-1 key-set, and the additive-vs-breaking rules; reference it from agent_contract_payload (cli.py:110). Update CLAUDE.md's 'read state first' guidance to mention contract_version. Record that 0.9.0 promises additive-only within a contract major, enforced by the Phase-3 gate. RISK: low — pure documentation; its value is that the fingerprint gate already makes the promise mechanical, so the doc describes an enforced rule rather than an aspirational one.

## 9. Locked decisions (confirmed 2026-07-21)

Decision 1 (**fix the frame before stamping** — accept the deliberate `ok`/failure-shape/`next_action` breaks) and decision 3 (**config too-new refusal scoped to the deploy/enqueue path only**; recovery + read-only stay permissive) were confirmed by the owner. Decisions 2, 4, 5, 6, 7, 8 are adopted as recommended below.

1. Do we fix the envelope BEFORE stamping contract 1 — normalize `ok` to a single meaning (adding a separate `health` field to doctor, keeping `result` as the graded outcome in run-batch, `removed` in hub-remove), add `next_action` to `status`, and collapse the three failure shapes into one `{ok:false,error{},next_action?}` envelope — accepting these as deliberate breaks against today's versionless shape? RECOMMEND YES. This is the single most important decision: #44 is the last free moment before the freeze, and stamping a number on today's overloaded `ok` (health-verdict at cli.py:569 / result==success at cli.py:649 / found-or-not at cli.py:1183 / absent on agent-contract) would notarize an interface agents already cannot rely on and force a painful 2.0. Version the fixed thing.
2. One global `CONTRACT_VERSION` int for all JSON/JSONL plus a separate `CONFIG_VERSION` int for the file — versus per-surface version namespaces? RECOMMEND one global int + one config int. Per-surface numbers nest and contradict inside embedded payloads (hub.py:198, status/inspect embed job dicts) and impose an N-way negotiation matrix; the per-surface proposal could not even hold its own line (it bucketed 13 payloads including the volatile recovery surface into one number).
3. Is the config too-new refusal scoped to the deploy/enqueue path ONLY, with `reconcile`/`recover`/`unlock` and all read-only commands allowed to proceed (warn + doctor.next_action)? RECOMMEND YES. Enforcing refuse-newer inside `load_config` would let a version mismatch after a rollback lock the operator out of the crash-recovery commands that are 0.3.0's flagship — recovery must be the most permissive surface, ship-code paths the most guarded.
4. Is the envelope key named `contract_version` (not bare `contract`)? RECOMMEND `contract_version`. Bare `contract` collides with the `agent-contract` command whose payload literally is the operating contract, guaranteeing agent confusion on the one surface where the word already means something else.
5. Do we SHIP a golden key-set fingerprint CI gate (tests/contract_fingerprints.json) that fails unless a payload shape change is matched by a version bump or a conscious golden-file regen? RECOMMEND YES — this is load-bearing. JSON has no physical forcing function like SQLite's user_version; without this gate the version number silently goes stale (the exact registry.py:85-112 write-without-check gap) and the 0.9.0 freeze promise is unbacked.
6. Do we DEFER the negotiation subcommand (`mergetrain contract --require x>=N`, new exit code 8) and expose the number only via `doctor`/`version` as the single source of truth? RECOMMEND DEFER. No consumer exists today that pins a major and self-refuses; building a two-party negotiation protocol before two parties disagree is YAGNI for a pre-1.0 stdlib tool.
7. Is the JSONL `stream_start` header re-emitted on EVERY connect including `--after` resume (records otherwise byte-identical)? RECOMMEND YES. A resume is a fresh, possibly different-binary process — the one moment a long-lived consumer must re-confirm the contract; the header carries no event id so id-based dedupe is unaffected.
8. Do we SHIP NO config-migration persist path in #44 (init writes version:1, existing files ride absent⇒1, in-memory migrate only)? RECOMMEND YES. Re-serializing hand-edited YAML from the frozen dataclass would destroy comments and silently-dropped unknown keys; at v1 there is nothing to persist, so the data-loss trap is simply not built.

## 10. Residual risks
- Phase 1 is a genuine breaking change to fields the hub dashboard and coding agents already parse (ok, error shape). If any consumer is not updated in lockstep it silently mis-reads outcomes. Mitigation: single reviewed PR, all goldens regenerated, migration note, done before any contract_version is published so nothing has pinned the old shape.
- The fingerprint gate cannot mechanically detect a same-keys value-TYPE or value-SEMANTIC change (e.g. next_action's set of values changing meaning). It catches key add/remove/rename only. Residual discipline for semantic changes still rests on review; the gate narrows but does not eliminate the human judgment surface.
- The snapshot boundary rule (build_dashboard_snapshot stays bare, only the HTTP handler and dump_json stamp) is a convention a future edit could violate by adding contract_version inside the builder, double-stamping the hub. The fingerprint test asserting nested snapshots are bare is the guard; if that assertion is dropped the convention silently rots.
- stream_start is a new first-line JSONL record; a naive existing consumer that assumed line 1 is always an event will mis-anchor. Justified as additive because id-based --after resume and the three existing record types are byte-identical, but it is technically a wire change shipped in the versioning PR — must be called out in the changelog and docs/contract.md.
- Config too-new enforcement is scoped to the deploy path, so a stale-but-newer config CAN run under recover/reconcile/unlock with only a warning. This is the deliberate recovery-permissive trade; the residual risk is an operator ignoring the warning and running recovery against a config whose semantics an old binary misreads. Accepted because locking recovery is the worse failure.
- Forward-only, no down-migration (inherited from store.py): once a file or the fingerprint set advances to contract 2, an older binary can only refuse the deploy path, never parse it. This one-way-door property is intentional but means a bad contract-2 release cannot be cleanly rolled back on the config/enqueue path without reverting the file.
